树（Tree）数据结构
第六章 数据库中的树
Database Trees: B-Tree & B+ Tree
多路平衡 · 磁盘优化 · 索引基石
从平衡树到多路平衡树
回顾
  • BST / AVL / 红黑树都是二叉树
  • 每个结点最多 2 个孩子
  • 平衡时树高 ≈ log₂n
问题
  • 数据量巨大时（千万级），树高仍有 20~30 层
  • 数据库数据存储在磁盘，每次访问一个结点 = 一次磁盘 IO
  • 二叉树太高 → IO 次数多 → 查询慢
解决思路
多路平衡树：一个结点存多个键，分多个支路，把树变"矮胖"
B-Tree 结构
m 阶 B-Tree 定义
  • 每个结点最多 m 个孩子，m−1 个键
  • 根结点至少有 2 个孩子（除非只有根）
  • 非根非叶结点至少有 ⌈m/2⌉ 个孩子
  • 所有叶子在同一层
阶 m = 3（2-3 树）：每个结点最多 2 个键，3 个孩子
核心思想：把磁盘页大小（如 16KB）作为一个结点，一次 IO 读入多个键
示例：3 阶 B-Tree

B+ Tree 结构
B+ Tree vs B-Tree
特性B-TreeB+ Tree
数据存储位置 所有结点 仅叶子结点
非叶子作用 存键+数据指针 纯索引
叶子连接 无 双向链表
范围查询 需中序遍历 链表顺序扫描
树高 略高 更低（索引更紧凑）
关键优势：非叶子结点不存数据，同样大小页能存更多键，树更矮

B+ Tree 示意图（重点：叶子链表）

为什么数据库选 B+ Tree？
优势说明
磁盘页对齐 每个结点正好一个磁盘页（如 16KB），一次 IO 满载
极低树高 3~4 层即可索引 千万级 数据（m 很大时）
范围查询友好 叶子链表顺序扫描，无需回溯
IO 次数稳定 每次查询都走到叶子，路径长度固定
空间局部性 叶子顺序存储，预读友好

知识衔接总结
核心思想演进
  • 二叉搜索 → 比较一次，排除一半
  • 多路搜索 → 比较一次，排除多路
  • 磁盘页优化 → IO 对齐，一次读一页
本章关键结论
  • B+ Tree 是关系型数据库索引的事实标准
  • 所有数据在叶子 + 叶子链表 = 范围查询神器
  • 树高 3~4 层 = 千万级数据仅需几次 IO
一句话总结：B+ Tree 把"二分搜索"的思想扩展到"多分搜索"，并用磁盘页对齐实现了海量数据的高效索引
